# 3. 1차 빌드의 핵심 목표

1차 빌드에서 가장 먼저 증명해야 하는 것은 기능 개수가 아니라 다음 핵심 경험이다.

```text
Caravan 준비
→ 도시와 무역로 선택
→ 상품·식량·마차·견인 동물·용병 구성
→ 무역 출발
→ 시간 경과와 이벤트
→ 도착 또는 실패
→ 수익·손실 정산
→ 성장 구매
→ 다음 무역에서 성장 효과 체감
```

## 성공 기준

- 준비부터 정산까지 한 사이클을 중단 없이 완료할 수 있다.
- 무역 결과가 다음 무역의 선택에 영향을 준다.
- 거리, 식량, 마차, 용병, 계절 또는 재난에 따라 결과가 달라진다.
- 성장 전후의 수치 차이가 실제 플레이에 반영된다.
- 게임 종료 후 다시 실행해 진행 상태가 복구된다.
- 신규 사용자가 튜토리얼을 통해 첫 무역을 완료할 수 있다.

---

## Caravan 개편 공통 원칙

- 기존에 상단으로 부르던 무역 운영 단위는 이후 `Caravan`으로 통일하며, 마차·견인 동물·무역품·식량·용병·무역 진행 상태를 포함한다.
- 마차의 `최대 적재량`은 물리적인 적재 상한이며, 이를 초과하면 출발할 수 없다.
- `최종 적정 적재량`은 이동속도 패널티가 시작되는 기준이며, 마차 기본 적정 적재량과 견인 동물의 추가 효율 적재량으로 계산한다.
- 견인 동물의 추가 효율 적재량은 최대 적재량을 증가시키지 않는다.
- 하나의 Caravan에는 한 종류의 견인 동물만 할당한다.
- 견인 동물 성장·등급은 비확정 사항이며 1차 빌드 필수 데이터와 마일스톤에서 제외한다.
- 식량은 경과한 인게임 시간에 비례해 소모하며, 온라인·오프라인 모두 같은 시간 배율 정책을 사용한다.
- 로드 이벤트 대응은 이벤트 ID 또는 태그 기반으로 연결하고, 마차가 무효화·강화한 결과를 정산에 기록한다.
- `최종 적정 적재량`, `과적 비율`, 최종 이동속도 등 계산 가능한 값은 원본 데이터로 중복 저장하지 않는다.

# 4. 구현 우선순위

| 우선순위 | 의미 | 팀 공통 판단 기준 |
|---|---|---|
| **P0** | 기반 필수 | 공용 데이터, 씬, 시간, 저장 인터페이스 |
| **P1** | 플레이 필수 | 무역 준비부터 정산·성장·재출발 |
| **P2** | 핵심 재미 검증 | 거리, 식량, 마차 슬롯, 적정 적재량, 과적, 견인 동물, 로드 이벤트, 경제 변화 |
| **P3** | 1차 빌드 완성 | 기부, 투자, 대출, 튜토리얼, 저피로도 UX |
| **P4** | 후순위 | 여러 Caravan, 고급 추천, 상세 연출, 콘텐츠 확장 |

## 운영 원칙

- P1 통합 빌드가 나오기 전에 P3 기능을 대규모로 구현하지 않는다.
- P0 공용 구조를 개인이 임의로 분기하지 않는다.
- P2와 P3는 완성형이 아니라 최소 1회 검증 가능한 형태로 구현한다.
- 일정이 지연되면 P4를 삭제하고 P3를 축소한다.
- 핵심 루프, 저장, 오프라인 진행은 삭제하지 않는다.

# 6. 마일스톤별 공통 통과 조건

## M0

- Boot → Title → Loading → InGame 흐름이 동작한다.
- 도시, 상품, Caravan, 마차, 견인 동물 더미 데이터가 로드된다.
- 마차 슬롯·기본 적정 적재량·최대 적재량과 견인 동물 추가 효율 적재량 계약이 공유된다.
- 공용 ID, 상태 enum, 정산 데이터 구조가 고정된다.
- 시간 배속과 무역 즉시 완료 디버그 기능이 있다.
- 모든 담당자가 같은 공용 구조를 사용한다.

## M1

- 무역 한 사이클을 3회 연속 완료한다.
- 슬롯 초과, 최대 적재량 초과와 필수 구성 누락이 출발 전에 차단된다.
- 최종 적정 적재량과 과적 비율이 동일한 입력에서 동일하게 계산된다.
- 하나의 Caravan에 한 종류의 견인 동물만 할당된다.
- 상품 구매와 판매가 가능하다.
- 정산 후 성장 구매가 가능하다.
- 성장 효과가 다음 무역에 반영된다.

## M2

- 단거리와 장거리의 결과가 다르다.
- 식량 부족 또는 약탈로 실패할 수 있다.
- 실패 후에도 게임을 계속할 수 있다.
- 마차·견인 동물·용병 선택이 결과에 영향을 준다.
- 과적 상태는 이동속도와 예상 식량 소모에 영향을 준다.
- 로드 이벤트에 대한 마차의 무효화·강화 효과가 실제 결과에 반영된다.
- 식량은 경과한 인게임 시간을 기준으로 소모된다.
- 계절 또는 재난이 가격에 영향을 준다.
- 정산 화면에서 보정과 손실 원인을 확인할 수 있다.

## M3

- 새 게임과 이어하기가 구분된다.
- 저장 후 재실행해 상태를 복구한다.
- 오프라인 중 완료된 무역이 정산 대기 상태가 된다.
- 오프라인 경과 시간도 동일한 인게임 시간 배율로 변환되어 진행도와 식량 소모에 반영된다.
- 저장 후 복구 시 최종 적정 적재량과 과적 비율은 원본 구성에서 다시 계산된다.
- 중복 보상이 발생하지 않는다.
- 기부, 투자, 대출이 최소 형태로 각각 1회 동작한다.

## M4

- 튜토리얼을 통해 첫 정산과 첫 성장을 완료한다.
- 튜토리얼 스킵 후 정상 플레이가 가능하다.
- 이전 무역 구성을 다시 불러올 수 있다.
- 예상 수익, 위험도, 권장 식량을 확인할 수 있다.
- 슬롯 사용량, 기본·추가·최종 적정 적재량, 최대 적재량, 과적 비율과 예상 속도 감소를 확인할 수 있다.
- 오후 6시 이후 신규 기능을 추가하지 않는다.

## M5

- Blocker와 Critical 버그가 없다.
- 새 게임, 이어하기, 성공, 실패, 오프라인 완료를 검증한다.
- 최소 3명이 독립적으로 빌드를 실행한다.
- 제출용 RC와 백업본을 확정한다.

# 11. 일일 협업 규칙

## 오전 공유

각 담당자는 다음만 공유한다.

1. 전날 통합된 결과
2. 오늘 통합할 결과
3. 막힌 인터페이스 또는 지원 요청

진척률은 퍼센트가 아니라 실행 가능한 결과로 표현한다.

## 매일 통합

- 최소 하루 1회 develop 브랜치에 통합한다.
- 통합 후 무역 한 사이클을 실행한다.
- 공용 구조 변경은 당일 공유한다.
- 기능 완료 후 장기간 개인 브랜치에 방치하지 않는다.

## 종료 전 기록

- 병합 커밋
- 현재 동작하는 기능
- 알려진 버그
- 재현 절차
- 다음 담당자에게 필요한 데이터
- 다음 날 최우선 작업

# 12. 브랜치 및 병합 규칙

```text
main
develop
```

# 13. 버그 우선순위

## Blocker

- 게임 실행 불가
- 씬 진입 불가
- 무역 시작 또는 정산 불가
- 저장 손상
- 무한 로딩
- 반복 크래시

## Critical

- 재화 무한 복사
- 중복 보상
- 성장 미적용
- 실패 후 진행 불가
- 필수 버튼 미작동
- 튜토리얼 진행 차단

## Major

- 가격 또는 정산 표시 오류
- 일부 이벤트 결과 누락
- 내구도 표시 불일치
- 비핵심 UI 깨짐

## Minor

- 오탈자
- 정렬
- 연출 부족
- 미세한 밸런스 문제

7월 15일에는 Blocker와 Critical만 최우선으로 수정한다.
