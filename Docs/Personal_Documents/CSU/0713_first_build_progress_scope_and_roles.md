## 1. 한 줄 요약

- **백엔드(시간·저장·무역·경제 계산)** 은 상당 부분 동작합니다.
- **플레이어가 버튼을 눌러 플레이하는 화면(UX)** 은 아직 거의 없습니다.
- 문서에 적힌 기능을 **전부** 넣기에는 시간이 부족합니다.
- 그래서 **최소 플레이 루프**만 목표로 줄이고, **화면 연결**에 인력을 모읍니다.
- 씬은 **한 번에 한 명만** 수정합니다. (충돌 방지)

## 2. 지금 프로젝트는 어디까지 왔나

### 2-1. 잘 되어 있는 것 (대략)

| 담당 | 상태 (쉬운 말) |
|------|----------------|
| **Framework** | 씬 흐름(Boot→Title→Loading→InGame), 인게임 시간·배율·Pause, JSON 저장, 정산 대기 저장, 오프라인 진행/완료, 디버그·Smoke 검증이 많이 갖춰짐 |
| **Core** | 무역 진행, Caravan(슬롯·과적·식량 등) 계산·검증이 상당 부분 있음 |
| **Progression / Economy** | 구매·판매·정산·성장 같은 **M1 경제 루프** 계산기가 있음 |
| **UI & Data** | 데이터 구조·화면 설계 문서·Sandbox 쪽 ViewData 등은 있음. **실제 InGame 화면 작업은 아직** |
| **Content / Art** | 아이콘·3D 등 리소스 유입. 최소 콘텐츠 ID를 UI용으로 “먼저 고정”하는 작업이 필요 |

### 2-2. 아직 약한 것

- **Title / InGame에서 손으로 플레이하는 UX** (가장 큰 빈칸)
- 기부 · 투자 · 대출 (폴더만 있거나 최소 구현 전 단계로 보임)
- 튜토리얼
- Framework의 **저장 안정성 계층**(임시·메인·백업) / 자동저장 Dirty 체계 — 일부는 미완, 1차에서는 축소 가능
- 팀 공통 최종 체크리스트는 아직 대부분 미체크 상태

### 2-3. 왜 UX가 특히 급한가

1차 빌드가 증명해야 하는 핵심은 기능 개수가 아니라 이 경험입니다.

```text
준비 → 출발 → 시간 경과 → 도착/실패 → 정산 → 성장 → 다시 준비
```

지금은 이 루프가 **디버그 메뉴 / Smoke**로는 많이 검증되어 있지만,  
**일반 플레이어용 버튼·패널**로는 아직 이어지지 않았습니다.

씬은 4개뿐입니다.

| 씬 | UX 부담 |
|----|---------|
| Boot | 거의 없음 |
| Loading | 거의 없음 |
| **Title** | 작음 (새 게임 / 이어하기 / 저장 초기 등) |
| **InGame** | **큼** (준비·진행·정산·성장이 한 씬에 모임) |

→ 실전 UX 전장은 **Title + InGame**, 그중 대부분은 **InGame**입니다.

## 3. 1차 빌드 축소 범위 (제안)

### 3-1. 이번 제출에서 꼭 보여 줄 것 (최소 목표)

```text
Title에서 새 게임 / 이어하기
→ InGame에서 무역 준비
→ 출발
→ 진행 화면
→ 정산 확인 후 수령(Claim)
→ (가능하면) 성장 1회
→ 저장 후 다시 켜서 이어하기
```

오프라인 완료·중복 보상 방지는 Framework 쪽에 이미 많이 있으므로, **깨지지 않게 유지**하는 것이 목표입니다.

### 3-2. 이번엔 미루거나 최소만 (2차 또는 동결 후 버그픽스만)

아래는 “없으면 제출 불가”가 아니라, **지금 새로 크게 만들지 말자**는 뜻입니다.

- 기부 · 투자 · 대출 UI/시스템 본구현
- 튜토리얼 · 마스코트 연출
- 예쁜 저피로도 UX, 고급 추천
- AtomicSave(임시·메인·백업) / 복잡한 AutoSave — 가능하면 축소 유지
- 콘텐츠 수량 대폭 확대, 상세 연출

> 팀 규칙상 “삭제 금지”에 튜토리얼 등이 있으나, **일정 상 전부 넣기 어려우면**  
> 제출 전 팀 합의로 **명시적 보류**를 결정하는 것이 안전합니다.  
> 이 문서는 그 합의를 위한 초안입니다.

### 3-3. 동결 이후 (7/14 18시~)

- **새 기능 · 구조 변경 병합 금지**
- 빌드 불가, 씬 진입 불가, 저장 유실, 중복 보상, 이어하기 불가 같은 **치명 버그만** 수정

## 4. 7월 16일까지 각자의 책임과 할 일

### 공통 규칙 (전원)

1. **같은 씬을 두 명이 동시에 열지 않습니다.**
2. 신규 메타 기능보다 **이미 있는 루프를 화면에 붙이는 일**을 우선합니다.
3. Prefab·스크립트는 씬 밖에서 만들고, **씬에 넣는 사람은 오너만** 합니다.
4. 막히면 “기능 추가”보다 **연결·버그**를 먼저 말합니다.

---

### UI & Data — InGame 씬 오너

**책임:** `InGame.unity`를 **혼자** 수정합니다. 플레이 화면의 최종 조립 담당입니다.

**할 일**

- 준비 / 진행 / 정산 / (여유 시) 성장 패널을 씬에 배치
- 버튼을 Framework · Core · Progression API에 연결
- Core/Progression/Art가 넘겨 준 Prefab을 받아 조립
- “한 사이클이 버튼만으로 도는지” 매일 확인

**하지 말 것**

- 경제 공식·무역 시뮬레이션을 UI 안에서 새로 계산하기
- Title 씬을 동시에 크게 수정하기 (Title은 Framework)

---

### Framework & Integration (CSU) — Title 씬 + 연결 지원

**책임:** `Title.unity` 수정. Adapter·이벤트·저장 경로에서 막히면 합류해 풉니다.

**할 일**

- Title: 새 게임 / 이어하기 / 필요 시 저장 초기 UI 연결
- 정산 Adapter, 화면 이벤트, Save/Load 이슈 대응
- InGame은 **직접 편집하지 않음** (요청 시에만 스크립트/연결 지원)
- Boot / Loading은 사실상 동결 (꼭 필요할 때만)

**하지 말 것**

- InGame 씬을 UI와 동시에 열기
- 동결 직전 AtomicSave 등 큰 신규 새로 시작하기 (제출 범위 밖이면 보류)

---

### Core Gameplay — Prefab · DTO · API 지원

**책임:** InGame에 넣을 **패널 Prefab**을 씬 밖에서 만들고, UI가 쓰는 **입력/결과 데이터 모양(DTO)** 을 깨지지 않게 고정합니다.

**할 일**

- 준비·진행 등에 필요한 Prefab을 UI에 넘기기 (씬 파일은 열지 않음)
- 표시·검증에 필요한 API를 안정적으로 유지
- UI/Framework가 요청한 버그·값 불일치 수정
- Adapter가 필요하면 **Framework에 도움 요청** (Core가 Adapter를 혼자 새로 설계하지 않음)

**하지 말 것**

- `InGame.unity` / `Title.unity` 직접 수정
- 기부·투자·대출 등 새 큰 시스템 추가 (합의된 최소 범위 밖)

---

### Progression / Economy — Prefab · DTO · API 지원

**책임:** 정산·성장 화면에 필요한 **결과 DTO 안정화**와 Prefab 병행 제작.

**할 일**

- 정산/성장 관련 Prefab을 씬 밖에서 제작해 UI에 납품
- 가격·정산·성장 결과 필드를 UI가 믿고 붙일 수 있게 유지
- Adapter 연결이 필요하면 Framework에 요청
- 표시용 숫자/항목이 빠지거나 바뀌면 UI에 즉시 공유

**하지 말 것**

- 씬 동시 편집
- 기부·투자·대출 본구현을 UI와 병렬로 크게 확장 (1차 보류 권장)

---

### Art / Content — 리소스 · 최소 데이터 먼저 고정

**책임:** 그림·아이콘·패널 아트를 공급하고, UI가 기다리지 않도록 **테스트용 ID/더미**를 빨리 확정합니다.

**할 일**

- 아이콘, 패널 스프라이트, 필요 Prefab 아트 공급
- **오늘~바로:** UI가 쓸 최소 목록을 팀에 공유  
  - 예: 도시 / 무역로 / 상품 / 마차 / 동물 **ID와 표시 이름**
  - 숫자는 나중에 바꿔도 됨. **ID와 이름만 먼저 잠그기**
- 이미 있는 Sandbox / SharedGameData 더미를 쓸지, 공식으로 선언

**하지 말 것**

- InGame/Title 씬을 UI·Framework와 동시에 열어 배치하기  
  → 리소스는 파일로 넘기고, 배치는 씬 오너가 함

#### “테스트 데이터가 늦으면 UI가 기다린다”는 뜻 (쉬운 말)

화면을 만들려면 “사과”, “도시 A” 같은 **보여줄 재료**가 필요합니다.  
그 재료(ID·이름)를 Content가 늦게 주면, UI는 버튼을 만들어도 연결할 대상이 없어서 멈춥니다.  
그래서 **완벽한 밸런스보다, 임시라도 확정된 목록**을 먼저 주는 것이 중요합니다.

## 5. 작업이 흐르는 그림

```text
Art/Content ──► 아이콘·문구·최소 ID/더미
Core/Progression ──► 패널 Prefab + DTO 고정
        │
        ▼
UI & Data ──► InGame 씬에 조립 · 버튼 연결
        │
Framework ──► Title 씬 + Adapter/이벤트/저장 이슈 지원
        │
        ▼
매일: 새게임 → 준비 → 출발 → 정산 Claim 이 버튼으로 되는가?
```

## 6. Prefab 우선순위 (제안)

시간이 없으니 아래 순서로 납품합니다.

1. **무역 준비** (출발까지)
2. **정산** (결과 보기 + Claim)
3. **진행 중** (진행률·간단 상태)
4. **성장** (시간 되면)

각 Prefab에는 짧게라도 적어 주세요.

- 어떤 버튼인지
- 누르면 어떤 API / 이벤트를 기대하는지

---

## 7. 팀에게 부탁하는 합의 사항

회의나 채팅에서 아래만 짧게 확인해 주세요.

1. 1차 제출 목표를 **최소 플레이 루프**로 줄인다. (동의 / 수정)
2. InGame 씬 오너는 **UI & Data**, Title 씬 오너는 **Framework**. (동의)
3. 기부·투자·대출·튜토리얼은 **1차 보류**할지, 최소 스텁만 넣을지. (결정 필요)
4. Content가 **최소 ID 목록**을 언제 공유할지. (날짜·담당)
