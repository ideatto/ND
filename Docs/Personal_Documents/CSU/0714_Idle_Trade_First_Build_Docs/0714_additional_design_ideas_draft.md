# 1. 마차 내구도 수리 및 수리 비용 시스템

**지위:** 2차 이후 설계 후보 (1차 통과조건 아님)

## 목적

마차 내구도를 단순 소모가 아니라 운영 비용이 따르는 관리 요소로 쓴다. 고성능·고레어리티 마차는 효율이 높은 대신 수리 비용도 커지게 하여, 상위 마차에 유지 비용과 경제적 리스크를 둔다.

## 기본 규칙

- 무역 또는 이벤트로 내구도가 감소할 수 있다.
- 감소분은 수리로 회복한다.
- 수리 비용은 회복하는 내구도 수치에 비례한다.
- 레어리티가 높을수록 내구도 1당 수리 비용 배율이 크다.
- 현재 내구도 = 최대 내구도면 수리 불가.
- 보유 재화가 수리 비용보다 적으면 수리 불가.

## 수리 비용 계산

기준 요소:

- 수리할 내구도 수치
- 내구도 1당 기본 수리 비용
- 레어리티별 수리 비용 배율

```text
최종 수리 비용
= 수리 내구도 × 내구도당 기본 수리 비용 × 레어리티 수리 비용 배율
```

## 필요 데이터 예시

- 최대 내구도 / 현재 내구도
- 내구도 1당 기본 수리 비용
- 마차 레어리티
- 레어리티별 수리 비용 배율 (개별 마차보다 공통 정책 데이터 우선 검토)

## 적용 시점

1차 빌드·7/16 제출 전 신규 구현하지 않는다. 팀 합의 후 2차·최종 제출 준비 기간에 우선순위를 정해 단계 구현한다. 1차에는 기존 내구도 데이터와 확정 무역 루프 UX·오류 수정만 한다.

# 2. 무역 중도 포기 및 무역 실패 패널티

**지위:** 2차 이후 설계 후보 (1차 통과조건 아님). 전자산 소실이므로 **팀 별도 합의 없이 기정사실로 취급하지 않는다.**

## 목적

출발 이후 결정에 위험을 두고, 무리한 출발·준비 부족 실패에 명확한 손실을 둔다.

## 적용 조건 (후보)

- 진행 중 무역을 중도 포기한 경우
- 진행 중 실패 조건이 충족된 경우
- 결과가 실패로 확정된 경우

## 패널티 규칙 (후보)

중도 포기 또는 실패 시, 해당 무역에 투입된 Caravan 구성을 **전부 소실**하는 방안을 후보로 둔다.

소실 대상 후보:

- 사용 중인 마차
- 배정된 견인 동물
- 적재된 무역품
- 고용된 용병
- 적재된 식량

해당 무역에 실제 배정된 자산만 대상으로 하고, 보관소·다른 Caravan에는 영향 주지 않는 방향을 검토한다.

## 처리 시점

실패·중도 포기 **확정 시 한 번만** 처리한다. 중복 호출·세이브 복구로 이중 차감되지 않도록 다음 기록 여부를 검토한다.

- 무역 ID
- 실패·포기 사유
- 패널티 처리 여부
- 소실 자산 목록
- 패널티 처리 시각

## UI 요구사항

중도 포기 전 확인 UI:

- 복구 불가 안내
- 소실 예정 자산 목록
- 최종 확인 / 취소

## 적용 시점

1차 범위·제출 전 신규 구현·저장 필드·데이터 계약 확장 금지. 팀 합의 후 2차 이후에 API·소실 처리·중복 방지·확인 UI를 단계 구현한다. 1차에는 기존 실패 정산·Claim·저장 복구 UX·디버깅만 한다.

# 3. 무역 출발 전 준비 취소 기능

**지위:** 2차 이후 설계 후보 (1차 통과조건 아님). 1차 Preparation 선택·출발 UX와 혼동하지 않는다.

## 목적

출발 확정 전이면, 배정한 마차·견인 동물·무역품·식량·용병을 안전하게 초기화할 수 있게 한다.

## 적용 범위

출발이 확정되기 전까지만 사용한다. 대상 예:

- 선택한 무역로 / 마차
- 배정한 견인 동물
- 적재한 무역품·식량
- 고용·배정한 용병

## 기능 규칙

- 준비 화면에 `무역 출발 취소` 또는 `준비 초기화` 제공
- UI만 지우지 않고 **준비 취소 전용 API** 호출
- 준비 중 무역 데이터 일괄 초기화
- 아직 확정 차감되지 않은 자산은 원래 상태로 복구
- 임시 예약·할당 해제
- 완료 후 준비 화면 기본 상태로 복귀

## API 분리

| 구분 | 의미 |
|---|---|
| 준비 취소 | 출발 전 임시 선택·할당 초기화 |
| 중도 포기 | 출발 후 진행 무역 종료 + 실패 패널티 |
| 무역 실패 | 규칙상 실패 + 실패 패널티 |

서로 다른 API 또는 명확히 구분된 처리 경로를 쓴다.

### 권장 API 예시

```csharp
bool TryCancelTradePreparation();
```

- `true`: 준비 초기화·할당 해제 성공
- `false`: 취소 불가 또는 초기화 실패

필요 시 실패 원인용 결과 객체·오류 코드를 검토한다.

## 적용 시점

1차·제출 전 신규 구현하지 않는다. 팀 합의 후 2차 이후에 `TryCancelTradePreparation()`·할당 해제·초기화 UI를 단계 구현한다. 1차에는 확정된 Preparation 선택·출발 UX·결함 수정만 한다.

# 4. UI 및 게임 데이터의 데이터 드리븐 관리

**지위:** 확정 기획이 아니다. UI·밸런스 수치를 코드 밖에서 관리할 수 있는지 **조사·소규모 시도(스파이크)** 한다. 전면 전환·공용 정책 에셋 대량 추가·하드코딩 일괄 제거는 본 문서만으로 확정하지 않는다. 시도 범위·담당·완료 기준은 별도 합의 후 마일스톤에 올린다.

## 탐색 목적

표시·콘텐츠 데이터를 외부 에셋으로 둘 수 있는지 알아본다. 검증 대상:

- 데이터 수정 시 코드 변경을 줄일 수 있는지
- UI·데이터 담당이 표시·밸런스를 상대적으로 독립적으로 다룰 수 있는지

## 조사·시도 후보 대상

아래는 **검토·소규모 시도** 후보이며, 전체 확정 적용 범위가 아니다.

- 도시·무역로·무역품·마차·견인 동물·용병 정보
- 마차 레어리티·수리 비용 정책
- UI 표시 이름·설명·아이콘·가격·수치
- 선택 목록·정렬 기준

## 시도 시 참고 원칙 (확정 계약 아님)

본 절만으로 공용 API·저장 계약을 바꾸지 않는다. 시도 시 참고:

- UI가 개별 SO·임시 데이터를 직접 탐색하지 않는 방향
- 공용 제공 인터페이스 또는 ViewData로 전달
- ID와 표시 이름 분리
- 원본 데이터와 UI 표시용 데이터 분리 여부 검토
- 목록 기반으로 콘텐츠 추가 시 UI 수정 최소화 가능성 확인
- 필수 데이터 누락 시 기본값·유효성 검사
- 필드 변경 시 UI·시스템 영향 확인

## 담당 논의 범위 (합의 전)

### UI & Data

- 필요 데이터 항목, SO·공용 에셋, ViewData, 아이콘·이름·설명·수치 연결, 누락·잘못된 참조 검증

### Framework & Integration

- 공용 제공 인터페이스, 로딩·완료 이벤트, 조회 API, 원본→ViewData 변환, 씬 진입 초기화 순서

### 각 기능 담당

- 소유 원본 데이터의 규칙·유효성 기준 제공

## 단계적 시도 순서 (제안)

1. 현재 UI가 쓰는 데이터 항목 식별
2. 하드코딩된 표시 데이터 일부만 외부로 분리 시도
3. 공용 조회 방식 맞춤 가능 여부 확인
4. 누락·중복 ID 검증
5. 소규모로 코드 수정 없이 콘텐츠 추가 가능한지 시험
6. 안정 시 적용 범위 확대를 **별도 합의**

## 적용 시점

1차·제출 전 전면 전환·유형 전환·정책 에셋 대량 추가·하드코딩 일괄 제거·무코드 콘텐츠 구조 구현 금지. 합의 전에도 본 절을 근거로 본격 전환에 착수하지 않는다.

1차에는 확정 Preparation·Traveling·Settlement UX용 기존 데이터 연결·누락 참조·표시·초기화 순서 디버깅만 한다. 조사·스파이크는 1차 제출 밖이며, 확대는 팀 합의 후 2차 이후 마일스톤에서 다룬다.
