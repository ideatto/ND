# 1. 게임 개요

플레이어가 거점에서 Caravan을 준비하고, 무역품과 식량을 적재하여 다른 도시로 출발한 뒤, 시간 경과에 따라 무역을 완료하고 정산하는 방치형 무역 경영 시뮬레이션이다.

장기적으로는 도시별 시장, 계절·재난, 로드 이벤트, 용병과 약탈, 성장, 투자 퀘스트, 구조 대출, 여러 Caravan 운영을 포함한다. 다만 1차 빌드에서는 해당 전체 기획을 구현하지 않고, **준비 → 이동 → 정산 → 재출발**이 실제 UI에서 끊기지 않는지를 검증한다.

# 2. 현재 프로젝트 상태

## 2.1 백엔드·디버그 경로 검증 완료

다음 항목은 2026-07-14 기준 **Framework API·디버그/Smoke 경로**로 동작을 확인했다. 플레이어 Title/InGame 버튼 UX와는 구분한다.

- Boot → Title → Loading → InGame → Title 씬 흐름

- FrameworkRoot 중복 생성 방지
- 무역 시작 시간·종료 예정 시간 기록
- Traveling 상태 저장 및 재실행 후 진행률 복구
- 오프라인 경과 시간 반영
- 정산 대기 상태와 정산 결과의 재실행 후 복구
- Claim 이후 재화 반영 및 중복 Claim 차단
- 성공 정산 후 다음 무역 재출발
- 실패 정산 후 Claim하고 다음 무역 재출발
- 무역 루프 3회 연속 실행
- 최대 적재량 초과 출발 차단
- 최종 적정 적재량 초과·최대 적재량 이하의 과적 출발 허용
- 서로 다른 견인 동물 혼합 출발 차단
- 최소 견인 동물 수 미달 출발 차단
- 인게임 시간 기준 식량 감소
- 식량 부족 상태 지속 후 실패 정산 진입
- 정산 수익·비용·순이익의 재화 반영

## 2.2 현재 진행 중 (플레이어 UX)

- 디버그 Context Menu가 아닌 플레이어용 Caravan 준비 UX
- 도시·무역로·상품·식량·마차·견인 동물 선택 UI
- 용병 고용 UI **단순 틀**(제출 루프 포함, 실기능·연결은 2차)
- 플레이어용 출발 버튼과 출발 불가 사유 표시
- Traveling 진행 화면
- Settlement 결과 화면과 Claim UX
- Title: SaveData 없을 때 Continue 비활성화, Reset SaveData 확인 창
- 전체 화면 연결, 레이아웃, 텍스트, 해상도 대응
- 최종 빌드 회귀 테스트와 Blocker 수정

## 2.3 확인된 결함 및 정리 대상

1. **Title Continue / Reset:** 제출용 제품 UX는 SaveData가 없으면 Continue를 비활성화하고, Reset SaveData는 확인 창으로 한 번 더 묻는다. 현재 빌드의 일부 Continue 동작은 **백엔드·디버그 검증용**이며, 제출 전 제품 UX로 맞춰야 한다.
2. 식량 부족으로 발생한 무역 실패가 정산에서 `WagonBroken`으로 표시될 가능성이 높다.
3. Traveling 중 내구도 0 실패는 **전투 이벤트 실패로 인한 내구도 감소**에서 발생할 수 있다. 해당 전투 이벤트는 아직 미구현이다. 일반 거리 마모만으로 1차 무역을 `WagonBroken` 실패 처리하지 않도록 정리한다.
4. `DraftAnimalData`의 최대 적재량 증가 값이 무시된다는 경고가 반복된다. 견인 동물은 최대 적재량이 아니라 최종 적정 적재량만 증가시켜야 한다.
5. 디버그 테스트용 Trade ID가 반복 사용된다. 1차 빌드 플레이 흐름에서는 런타임 무역 단위를 안전하게 식별해야 한다.
6. 플레이어 UX가 완성되기 전까지 핵심 기능은 디버그 도구를 통해서만 조작 가능하다.

# 3. 축소된 1차 빌드 목표

## 3.1 한 줄 목표

플레이어가 별도의 개발자 도구 없이 다음 흐름을 한 번 이상 완료하고, 저장 후 재실행해 같은 상태를 이어갈 수 있는 빌드를 제출한다.

```text
새 게임 또는 이어하기
→ Caravan 준비
→ 무역 출발
→ 이동 진행
→ 성공 또는 실패 정산
→ Claim
→ 다음 무역 준비
```

## 3.2 1차 빌드 포함 범위

### 씬과 시작 흐름

- Boot, Title, Loading, InGame
- 새 게임
- SaveData가 있을 때만 Continue 활성·정상 이어하기
- SaveData가 없을 때 Continue 비활성
- Reset SaveData 시 확인 창으로 한 번 확인
- InGame에서 Title 복귀
- 화면 전환 중복 입력 방지

### Caravan 준비

- 운영 가능한 Caravan 1개
- 시연 가능한 도시·무역로 최소 1세트
- 무역품과 식량 적재
- 마차 선택
- 견인 동물 종류와 수 선택
- 용병 고용 UI **단순화된 틀**(화면·흐름만 제출 루프에 포함)
- 현재 적재량 및 적재 한도 표시
- 출발 가능 여부와 차단 사유 표시

기존에 제작된 추가 도시·상품·마차·동물 데이터는 사용할 수 있으나, 1차 빌드 통과 조건을 콘텐츠 수량 확대로 잡지 않는다.

### 출발 검증

- 필수 구성 누락 차단
- 슬롯 또는 중첩 한도 검증
- 최대 적재량 초과 차단
- 최종 적정 적재량 초과·최대 적재량 이하 과적 허용
- 과적 상태 및 속도 영향 반영
- 한 Caravan에 서로 다른 견인 동물 종류 혼합 차단
- 마차의 최소 견인 동물 수 미달 차단

### 이동과 식량

- 무역 시작 시간과 종료 예정 시간 기록
- 인게임 시간 기준 진행률
- 인게임 시간 기준 식량 소모
- 식량 부족 시 속도 저하 로직을 **구현하되**, 제출 빌드에서 적용 여부는 **설정(플래그)으로 선택** 가능해야 한다. QA 결과에 따라 On/Off를 결정한다.
- 식량 부족 상태가 허용 시간 이상 지속되면 실패
- 온라인과 오프라인에 같은 시간 변환 정책 적용
- 재실행 후 Traveling 상태와 진행률 복구

### 정산과 반복

- 성공 또는 실패 결과 생성
- 구매 비용, 판매 수익, 총비용, 순이익 표시
- 실패 원인 표시
- Claim 시 재화 반영
- 동일 정산 중복 Claim 방지
- 정산 대기 중 종료해도 결과 복구
- Claim 후 Preparation 복귀
- 성공과 실패 이후 모두 다음 무역 가능

### 저장

- 새 게임 기본 데이터
- JSON 저장과 불러오기
- 무역 출발·진행·정산 대기·Claim 결과 저장
- 오프라인 진행 복구
- 저장된 Caravan 원본 구성 복구
- 파생값은 복구 후 재계산

고급 원자적 저장, 다중 복구 후보, HMAC·암호화는 1차 빌드 제출의 통과 조건으로 사용하지 않는다. 이미 구현된 부분은 유지하되, 제출 전 신규 확장 작업은 하지 않는다.

### 최소 UX

- 플레이어가 디버그 Context Menu 없이 무역을 준비하고 출발할 수 있다.
- Preparation, Traveling, Settlement 상태를 구분할 수 있다.
- 잘못된 구성의 이유를 화면에서 확인할 수 있다.
- 정산 결과와 Claim 가능 여부를 화면에서 확인할 수 있다.
- 필수 버튼이 목표 해상도에서 정상 작동한다.

# 5. 핵심 규칙

## 5.1 Caravan

Caravan은 다음 원본 상태를 가진 무역 운영 단위다.

- 마차
- 견인 동물
- 무역품
- 식량
- 현재 무역 진행 상태

용병 **고용 UI 틀**은 1차 제출 루프에 포함한다. 고용 수치·전투·약탈 등 **실기능과 UI 연결은 2차 빌드**에서 진행한다.

## 5.2 적재량

```text
최종 적정 적재량
= Min(
    마차 기본 적정 적재량
    + 견인 동물 1마리당 추가 효율 적재량 × 배치 수,
    마차 최대 적재량
)
```

견인 동물은 마차의 물리적 최대 적재량을 증가시키지 않는다.

| 상태 | 조건 | 결과 |
|---|---|---|
| 적정 적재 | 현재 적재량 ≤ 최종 적정 적재량 | 속도 패널티 없음 |
| 과적 | 최종 적정 적재량 < 현재 적재량 ≤ 최대 적재량 | 출발 가능, 속도 감소 |
| 적재 불가 | 현재 적재량 > 최대 적재량 | 출발 차단 |

## 5.3 견인 동물

- 한 Caravan에는 한 종류만 배치한다.
- 마차의 최소 필요 수를 충족해야 한다.
- 이동속도, 추가 효율 적재량, 식량 소모에 영향을 준다.
- 견인 동물의 성장·등급은 1차 빌드에서 사용하지 않는다.

## 5.4 식량 부족

```text
정상 이동
→ 식량 0
→ 식량 부족 상태
→ (옵션 On일 때) 이동속도 감소
→ 허용 시간 내 해결되지 않음
→ 무역 실패
→ 실패 정산
```

식량이 0이 되는 순간 즉시 실패하지 않는다. 실패 결과에는 식량 부족을 나타내는 별도 원인이 기록되어야 하며, 일반 실패를 `WagonBroken`으로 일괄 표시하지 않는다.

**식량 부족 시 이동속도 감소**는 Core에 구현한다. 다만 1차 제출에서 실제로 적용할지는 **런타임/설정 플래그로 On·Off** 할 수 있어야 하며, QA 후 팀 결정으로 켠다. 유예 후 실패 자체는 1차 통과 조건에 포함한다.

## 5.5 마차 내구도

축소된 1차 빌드에서 마차 내구도는 **일반 거리 마모만으로 Traveling 실패를 만들지 않는다.**

- 정상 무역 성공 시 정해진 양을 감소시키는 방향은 유지할 수 있다.
- Traveling 중 내구도 0이 되어 `WagonBroken` 실패로 이어지는 경우는, **아직 미구현인 전투 이벤트 실패로 내구도가 깎일 때**를 전제로 한다.
- 1차에서는 전투 이벤트가 없으므로, 식량 실패 등이 `WagonBroken`으로 오인·덮어쓰이지 않게 원인 분리한다.
- 내구도 0 마차의 출발 제한 등은 후속 정책으로 확정한다.

## 5.6 경제 정산

```text
순이익 = 판매 수익 - 구매 비용 - 적용된 기타 비용
```

정산 화면은 최소한 다음 값을 표시한다.

- 판매 수익
- 총비용
- 순이익
- 성공 또는 실패
- 실패 원인

성장, 계절, 재난, 기부, 투자, 대출 보정은 축소된 1차 빌드 **제출 경로·통과 공식**에 포함하지 않는다. 성장 관련 코드(`GrowthCalculator`, `ApplyGrowthPurchase` 등)가 저장소에 있어도, 1차 플레이어 루프와 기본 정산에 섞이지 않게 동결·우회한다.

## 5.7 씬 소유권 (SceneOwners)

기본은 **기존 담당 소유권**으로 진행한다. `InGame=UI / Title=Framework` 분담은 1차 제출을 위한 **일시적 비상책**이었다.

- 평상시: 기존 SceneOwners 규칙을 따른다.
- 일정·충돌이 있을 때만 **협의 후 일시적으로** 씬 소유권자를 바꿀 수 있다.
- 같은 씬을 두 명이 동시에 수정하지 않는다.

# 6. 1차 빌드 통과 조건

## 플레이 흐름

- 새 게임에서 플레이어용 UI로 무역을 시작할 수 있다.
- SaveData가 있을 때 Continue로 이어하기가 가능하다.

- Preparation → (용병 고용 UI 틀 포함 가능) → Traveling → Settlement → Preparation을 완료한다.
- 성공 후 재출발할 수 있다.
- 실패 후 Claim하고 재출발할 수 있다.

## 데이터와 계산

- 적재·과적·견인 동물 검증 결과가 실제 출발 가능 여부와 일치한다.
- 식량이 인게임 경과 시간에 따라 감소한다.
- 식량 부족 유예 후 실패가 발생한다.
- 식량 부족 속도 저하 로직은 구현되어 있고, 제출 적용 On/Off는 QA·팀 결정으로 선택한다.
- 실패 원인이 실제 원인과 일치한다. 식량 실패가 `WagonBroken`으로 표시되지 않는다.
- 정산의 수익·비용·순이익과 재화 변화가 일치한다.
- 1차 제출 경로 정산에 성장·메타 보정이 섞이지 않는다.

## 저장과 복구

- Traveling 상태에서 종료 후 진행률이 복구된다.
- SettlementPending 상태에서 종료 후 같은 정산이 복구된다.
- Claim한 정산이 재실행 후 다시 나타나지 않는다.
- 동일 정산이 두 번 지급되지 않는다.

## 품질

- 무역 루프를 3회 연속 실행한다.
- Console Error가 없다.
- 플레이를 막는 Warning이 없다.
- 핵심 버튼이 중복 실행되지 않는다.
- 목표 해상도에서 필수 UI가 잘리거나 겹치지 않는다.

- 2026-08-05 이후 신규 기능 추가를 금지한다.

- 최종 빌드 기간에는 신규 대형 시스템을 구현하지 않는다.

# 9. 2차 빌드 목표와 범위

## 9.1 한 줄 목표

플레이어가 제품 UI만으로 무역을 준비하고, 시간과 위험을 관리하며, 정산·성장·투자 퀘스트·구조 대출을 거쳐 다음 무역 전략을 선택할 수 있는 전체 플레이 루프를 완성한다.

```text
Caravan 준비
→ 도시·무역로·상품·식량·마차·견인 동물·용병 선택
→ 출발 검증 및 저장
→ Traveling
→ 계절·재난·로드 이벤트·약탈
→ 성공 또는 실패 정산
→ 선택적 구조 대출 상환
→ Claim
→ 플레이어·Caravan·거점 성장
→ 투자 퀘스트 및 콘텐츠 해금
→ 파산 판정 또는 구조 대출
→ 다음 무역
```

## 9.2 2차 빌드 필수 범위

### A. 플레이어용 무역 준비

- 도시와 목적지 선택
- 무역로 선택
- 상품 구매와 적재
- 식량 구매와 적재
- 마차 선택
- 견인 동물 종류와 수 선택
- 용병 선택과 고용
- 슬롯, 중첩, 현재 적재량 표시
- 최종 적정 적재량과 최대 적재량 표시
- 과적 여부와 속도 영향 표시
- 예상 이동 시간
- 예상 식량 소모
- 예상 수익과 위험도
- 출발 불가 사유
- 출발 전 준비 취소 또는 초기화
- Caravan별 이전 무역 구성 불러오기
- 중요 출발 Command의 저장 성공 여부에 따른 화면 전환

### B. Traveling

- 진행률
- 남은 시간
- 현재 이동속도
- 남은 식량
- 현재 계절과 재난
- 적용된 이동·가격·위험 보정
- 로드 이벤트 결과
- 약탈 이벤트 결과
- 온라인·오프라인 동일 시간 정책
- Traveling 상태 저장과 재실행 복구
- 마차 내구도 0 도달 시 즉시 실패 확정

### C. 정산과 반복

- 성공 정산
- 식량 부족 실패
- 약탈 또는 전투 실패
- 마차 파괴 실패
- 구매 비용
- 판매 수익
- 식량 비용
- 용병 비용
- 이벤트 보상과 손실
- 이번 정산의 양의 거래 재화 지급액
- 순이익
- 발전용 재화
- 실패 원인
- 구조 대출 상환 선택
- Claim
- 중복 Claim 차단
- SettlementPending 복구
- 성공·실패 후 재출발

정산 Claim과 구조 대출 상환은 하나의 staged transaction으로 처리한다. 저장 실패 시 재화, 대출 원금, Trade 상태, PendingSettlement, Caravan 준비 데이터 등 해당 Command가 변경한 상태를 실행 직전 상태로 복구한다.

### D. 성장

2차 빌드에서 최소 다음 성장 항목을 구현한다.

- 플레이어 성장 1종 이상
- Caravan 성장 1종 이상
- 발전용 재화
- 성장 비용
- 성장 전후 비교
- 성장 상태 저장
- 성장 효과의 다음 무역 반영
- 구조 대출 제한 모드에서는 성장 구매 차단

성장축의 세부 확장은 후속 밸런스 대상으로 두되, 플레이어 성장과 Caravan 성장이 동일한 역할을 중복하지 않도록 한다.

### E. 마차 내구도·파괴·수리

#### 내구도와 파괴

- 정상 무역과 이벤트 결과에 따른 내구도 감소
- 내구도는 0 미만으로 내려가지 않도록 clamp
- 현재·최대 내구도 표시
- 내구도가 0이 되면 마차 파괴
- 파괴된 마차는 소유 데이터에서 삭제하고 수리 불가
- Caravan과 준비·재사용 구성의 마차 참조 제거
- 마차에 적재된 무역품과 식량 전부 소실
- Traveling 중 파괴 시 해당 무역을 즉시 `Failed`로 확정
- 마차 파괴, 참조 제거, 적재물 손실, Trade 실패, 정산 snapshot 생성을 하나의 staged operation으로 처리

정산 결과에는 최소한 다음을 포함한다.

- 마차 파괴 여부
- 파괴된 마차 ID
- 손실된 무역품
- 손실된 식량
- 실패 사유
- `caravanId`
- 전체 `tradeId`

#### 수리

```text
rawCost
= repairedDurability
× repairCostPerDurability
× wagonRarityMultiplier

finalCost = floor(rawCost)
```

- 모든 배율을 적용한 뒤 최종 결과에서 한 번만 내림
- 수리량이 1 이상이고 계산 결과가 0이면 최소 비용 1
- 수리 비용은 거래 재화를 사용
- 최대 내구도를 초과하는 수리 차단
- 내구도 0으로 파괴된 마차는 수리 불가
- 허용 수리량은 `1 ≤ 수리량 ≤ 최대 내구도 - 현재 내구도`
- 수리 단가와 레어리티 배율은 Shared Data에서 제공
- 수리 Command는 저장 성공 후에만 완료 이벤트와 UI 성공 상태를 확정

### F. 거점 성장·건축 재료·건물 업그레이드

#### 건축 재료

건축 재료는 통합 숫자 재화가 아니라 실제 아이템이다.

```text
무역 도시에서 Caravan이 건축 재료 구매
→ 거점 도시로 귀환
→ 플레이어가 이전할 물품과 수량 선택
→ 거점 도시 인벤토리에 저장
→ 건물 업그레이드 비용으로 사용
```

- 건물 업그레이드는 거점 도시 인벤토리의 물품만 사용
- Caravan에 남은 물품은 직접 사용할 수 없음
- Caravan과 거점 인벤토리 변경을 함께 저장
- 중복 요청 합계, 적재 수량, 이전 가능 상태, 인벤토리 용량을 검증

#### 건물 업그레이드

- 거점 건물 또는 시설 1종 이상
- 건물별·레벨별 요구 아이템
- 건물 효과
- 건물 효과의 무역 준비·정산 반영
- 안정적인 `buildingId`와 현재 레벨 저장
- 표시 이름과 비용 정의는 Shared Data에서 조회
- 비용, 레벨 증가, 파생 상태 재구성, 저장을 하나의 transaction으로 처리
- 구조 대출 제한 모드에서는 건물 업그레이드 차단

목표 저장 구조:

```csharp
[Serializable]
public sealed class VillageBuildingSaveData
{
    public string buildingId;
    public int level;
}
```

기존 저장이 표시 이름을 식별자로 사용한다면 migration 또는 compatibility adapter가 필요하다. 문서 수정만으로 SaveData 버전을 변경하지 않는다.

### G. 계절과 재난

2차 빌드 최소 범위:

- 여름
- 겨울
- 가뭄
- 홍수
- 도시별 상품 가격 또는 수량 변화
- 이동속도·식량·위험도 보정 중 필요한 항목
- 가격 변화 원인 표시

봄과 가을 및 추가 재난은 콘텐츠 여력에 따라 포함할 수 있으나 2차 필수 통과 조건으로 사용하지 않는다.

### H. 로드 이벤트와 용병·약탈

- 긍정 로드 이벤트 2종 이상
- 부정 로드 이벤트 2종 이상
- 약탈 이벤트 1종 이상
- 용병 전투력
- 적 전투력
- 성공·실패 판정
- 상품·식량·내구도 손실은 0까지 허용
- 모든 수량과 내구도는 0 미만으로 내려가지 않도록 clamp
- 이벤트 결과의 Settlement 기록
- 한 무역에서의 손실 상한은 별도 보호 최소값 없이 0까지 허용

용병 소멸과 견인 동물 손실은 일정에 따라 제외할 수 있으나, 용병 고용 비용과 약탈 대응 효과는 실제 플레이에 반영되어야 한다.

### I. 독립 기부 시스템 제거

기존 독립 기부 시스템은 2차 빌드 구현 대상에서 제거한다.

삭제되는 목표 정책:

- 도시별 기부 잔액
- 누적 기부금
- 기부 단계와 자연 감소
- 감소 주기·감소량·오프라인 상한·보호 최소값
- 기부 이벤트 소비
- 기부금을 투자 진행도로 전환하는 구조

`TownDonationSaveData`가 Production에서 아직 사용되지 않는다면 목표 SaveData 계약에서 제거한다. 이미 구현 또는 저장된 데이터가 있다면 즉시 삭제하지 않고 compatibility 정책을 둔다.

### J. 투자 퀘스트

기존 누적 기부·투자 진행도 시스템을 **일시불 투자 퀘스트**로 대체한다.

```text
투자 퀘스트 선택
→ 요구 비용 확인
→ 거래 재화와 Caravan 무역품을 한 번에 제출
→ 완료·해금·저장
→ 저장 성공 후 이벤트와 UI 갱신
```

기본 규칙:

- 누적 기부금 없음
- 투자 진행도 없음
- 분할 납부 없음
- 동일 퀘스트 중복 완료 및 중복 보상 금지
- 비용 정의는 Shared Data 또는 ScriptableObject가 소유
- SaveData에는 정의값을 복사하지 않고 완료 결과만 저장
- 보상은 완료와 동시에 적용하며 별도 Claim을 기본적으로 사용하지 않음

제출 가능한 자원:

- 거래 재화
- Caravan이 현재 직접 보유하거나 적재한 무역품

제출 불가능한 자원:

- 거점 도시 인벤토리의 무역품
- 건축 재료
- 승인되지 않은 퀘스트 전용 아이템
- Traveling 또는 SettlementPending 상태의 Caravan에 잠긴 물품

여러 Caravan에서 물품을 제출할 수 있으나, 각 제출에는 정확한 `caravanId`, `itemId`, `amount`가 필요하다. 비용 차감, Caravan별 물품 차감, 퀘스트 완료, 해금, 저장을 하나의 transaction으로 처리하고 저장 실패 시 모두 rollback한다.

목표 저장 의미:

```csharp
[Serializable]
public sealed class InvestmentQuestSaveData
{
    public string investmentQuestId;
    public string townId;
    public bool isCompleted;
    public long completedUtcTicks;
}
```

### K. 파산과 구조 대출

#### 기본 계약

구조 대출은 진행 불능 방지를 위한 단일 구제 시스템이다.

- 무이자 고정 기준 원금 모델
- 연체, 신용 점수, 벌금, 이자 없음
- 한 번에 하나의 활성 구조 대출만 허용
- 현재 세이브에서 한 번만 사용 가능
- 활성 상태와 과거 사용 여부를 분리 저장
- 제한 모드 종료와 원금 상환 완료는 별개

#### 지급 기준

Shared Data에 고정된 구조용 최소 무역 구성을 정의한다.

- 기준 마차
- 기준 견인동물 종류와 필요 수
- 필수 용병
- 식량 수량
- 무역품 종류와 수량
- 기준 무역로
- 구성 요소별 구조 대출 기준 가격

대출액은 총자산의 단순 합산액이 아니라, 현재 사용 가능한 자산과 기준 구성의 각 필수 요소를 비교하여 **실제로 부족한 요소의 기준 비용 합계**로 계산한다.

사용 가능한 자산:

- 거래 재화
- 사용 가능한 마차
- 배치 가능한 견인동물
- 배치 가능한 용병
- 사용할 수 있는 식량과 무역품

제외되는 자산:

- 파괴되거나 사용할 수 없는 마차
- 다른 진행 중 무역에 잠긴 자산
- 퀘스트 전용 물품
- 건축 전용 재료
- 기준 구조 구성에 사용할 수 없는 물품

#### 제한 모드

대출 발급 직후 `RestrictedPreparation` 상태에 진입한다.

허용:

- 구조 무역 준비
- 필요한 마차·견인동물·용병·식량·무역품 구매와 배치
- Caravan 구성 변경
- 준비 취소
- Title 이동
- 설정
- 게임 종료

차단:

- 성장 구매
- 건물 업그레이드
- 투자 퀘스트 완료
- 구조 무역과 무관한 경제 행동
- 구조 대출금을 다른 목적으로 사용하는 행동

Title 이동 또는 재실행 후에도 제한 상태를 유지한다. 첫 구조 무역의 출발과 해당 상태 저장이 모두 성공한 시점에 제한 모드를 해제한다. 원금이 남아 있다면 대출 활성 상태는 계속 유지한다.

#### 상환

구조 대출은 정산 시 선택적으로 상환한다.

- 선택지: `상환하기`, `상환하지 않기`
- 부분 금액 직접 입력 없음
- 이번 정산의 양의 거래 재화 지급액이 0 이하이면 상환 선택 미표시

```text
상환 가능액
= min(이번 정산의 양의 거래 재화 지급액, 남은 대출 원금)
```

`상환하기` 선택 시 상환 가능액 전액을 상환한다. `상환하지 않기` 선택 시 이번 정산에서는 원금을 차감하지 않는다. 출발 전에 이미 차감된 구매·식량 비용을 정산 시 다시 차감하지 않는다.

#### 재파산과 게임오버

구조 대출을 사용한 뒤 다음 조건을 모두 만족하면 추가 대출 없이 게임오버로 처리한다.

1. 현재 거래 재화로 최소 구조 구성을 만들 수 없음
2. 사용 가능한 보유 자산을 포함해도 유효한 무역 구성을 만들 수 없음
3. Traveling 상태 무역이 없음
4. Claim 가능한 PendingSettlement가 없음
5. 구조 대출을 이미 사용함

다른 Caravan이 무역 중이거나 수령 가능한 정산이 있으면 즉시 게임오버로 처리하지 않는다.

- 첫 정산 안내

- 투자 퀘스트 안내

- 권장 식량

마스코트는 안내 문구와 최소 연출에 사용할 수 있으나, 복잡한 시나리오·대화 시스템은 필수 범위가 아니다.

### M. 저장·복구·Command 결과

#### 저장 범위

- Traveling 저장·복구
- SettlementPending 저장·복구
- 성장 저장
- 마차 내구도와 파괴 결과 저장
- 건물 레벨과 거점 인벤토리 저장
- 계절·재난 저장
- 투자 퀘스트 완료와 해금 저장
- 구조 대출 원금·단계·과거 사용 여부 저장
- 이전 구성 저장
- 중요 상태 즉시 저장
- 데이터 변경 시 Dirty 자동저장
- 누락 필드 기본값 정규화
- 저장 버전 기록

#### SaveResult 목표 계약

팀의 승인된 목표 방향은 다음과 같다.

```csharp
SaveResult Save(SaveData data);
```

초기 Progression Command는 `SaveResult`를 직접 반환할 수 있다.

```csharp
SaveResult RepairWagon(...);
SaveResult UpgradeBuilding(...);
SaveResult CompleteInvestmentQuest(...);
SaveResult IssueRescueLoan(...);
```

초기 `SaveResult`는 저장 결과와 Command 사전 검증 거부를 함께 표현하는 임시 통합 결과다. 기능별 payload 또는 세분화된 도메인 오류가 필요해지면 장기적으로 `CommandResult`가 `SaveResult`를 포함하는 구조로 확장한다.

#### 저장 실패 UI

- `SaveResult.Message`는 개발자 로그 전용
- 내부 예외와 파일 경로를 플레이어에게 직접 표시하지 않음
- UI는 실패 이유를 사용자 메시지 키로 변환
- 저장 실패 시 성공 메시지, 완료 이벤트, 다음 화면 전환 금지
- 저장 처리 중 해당 기능 패널 입력 차단
- rollback 성공 후 ViewData 재조회
- rollback 실패 시 해당 입력을 계속 차단하고 Title 복귀 또는 재시작 안내

#### Snapshot·Rollback

세 가지 복구 데이터를 구분한다.

| 구분 | 생성·갱신 시점 | 목적 |
|---|---|---|
| `PreCommandSnapshot` | 중요 Command 실행 직전 | 해당 Command 저장 실패 시 복구 |
| `LastDurableSnapshot` | 저장 성공 후 | 현재 세션 전체 복구 |
| Disk Backup | 메인 저장 교체 전 | 저장 파일 손상 또는 로드 실패 복구 |

SaveData deep copy는 동일한 직렬화 경로를 사용한다. 복사 후 정규화, 검증, 런타임 lookup 재구성을 수행한다. 파생값과 ViewData는 snapshot에 저장하지 않는다.

재시도 정책:

- 최초 저장 포함 최대 3회
- 1차 실패 후 0.2초
- 2차 실패 후 0.5초
- 일시적 파일 잠금·쓰기·교체 실패만 재시도
- 데이터·검증·직렬화·snapshot 실패는 재시도하지 않음
- 최종 실패 후 `PreCommandSnapshot`으로 rollback
- rollback 실패는 별도 최종 실패로 처리

동시성 정책:

- 중요 Command 실행 중 다른 중요 Command 입력 차단
- 중요 저장 Command끼리 병합하지 않음
- rollback 완료 전 다음 중요 Command 실행 금지
- Dirty autosave는 중요 저장 완료 후 실행
- 중복 Dirty autosave만 병합 가능

## 9.3 멀티 Caravan 유지 정책

- 실제 게임 규칙상 최대 Caravan 수는 4개
- SaveData 컬렉션 크기를 4로 고정하지 않음
- 최대 수 제한은 Progression/System 또는 Shared Data가 소유
- 각 Caravan은 독립된 준비·Traveling·SettlementPending 상태를 가질 수 있음
- 무역과 정산의 키는 `caravanId + full tradeId`
- 투자 퀘스트의 Caravan 물품 제출은 정확한 `caravanId`를 요구
- Traveling 또는 SettlementPending Caravan의 자산 사용은 Command 검증에서 차단
- 마지막 선택 Caravan과 Caravan별 미완성 준비 구성을 저장

# 10. 2차 빌드 통과 조건

## 10.1 플레이 흐름

- 플레이어 UI만으로 유효한 Caravan을 구성하고 출발한다.
- 무역 준비부터 정산, 성장, 다음 무역까지 끊기지 않는다.
- 성공·실패·오프라인 완료를 각각 처리한다.
- 투자 퀘스트를 최소 1회 완료하고 해금 보상을 사용할 수 있다.
- 구조 대출을 발급받아 제한 모드에서 구조 무역을 준비하고 출발할 수 있다.
- 구조 무역 출발 저장 성공 후 제한 모드가 해제된다.
- 정산 시 대출 상환 여부를 선택할 수 있다.
- 구조 대출 사용 후 재파산 조건을 만족하면 게임오버가 발생한다.

## 10.2 계산과 데이터

- 슬롯·적재·과적·속도·식량 계산이 UI 표시와 일치한다.
- 가격 변화 원인을 확인할 수 있다.
- 계절·재난·이벤트 보정이 중복 적용되지 않는다.
- 정산 결과와 실제 재화 변화가 일치한다.
- 성장과 건물 효과가 다음 무역에 반영된다.
- 수리 비용은 최종 배율 적용 후 한 번만 내림 처리한다.
- 마차 파괴 시 참조와 적재물이 일관되게 제거된다.
- 투자 퀘스트가 거점 도시 인벤토리 물품을 차감하지 않는다.
- 구조 대출액은 실제 결손 요소의 기준 비용만 포함한다.
- 실패 후에도 구조 대출 사용 전에는 복구 가능한 경로가 존재한다.

## 10.3 저장과 복구

- Traveling에서 재실행 후 정상 복구한다.
- SettlementPending에서 같은 결과를 복구한다.
- Claim 완료 상태가 다시 지급되지 않는다.
- 성장·건물·투자 퀘스트·구조 대출이 재실행 후 유지된다.
- 해금된 도시·경로·기능이 재실행 후 유지된다.
- 대출 제한 모드가 Title 이동과 재실행 후 유지된다.
- 저장 실패 시 해당 Command의 재화·아이템·상태·해금이 rollback된다.
- 저장 실패 시 성공 이벤트와 화면 전환이 발생하지 않는다.
- SaveData 누락 필드가 안전하게 기본값으로 보정된다.

## 10.4 콘텐츠와 UX

- 최종 목표 콘텐츠의 70~80% 이상이 입력되어 있다.
- 신규 사용자가 설명서 없이 첫 무역과 첫 성장을 완료한다.
- 출발 불가 사유, 실패 원인, 가격 변화 원인이 표시된다.
- 투자 퀘스트 제출 출처와 수량을 확인할 수 있다.
- 구조 대출 제한 상태, 남은 원금, 상환 결과를 확인할 수 있다.
- 저장 실패 메시지가 내부 예외나 파일 경로를 노출하지 않는다.
- 목표 해상도에서 핵심 UI가 잘리거나 겹치지 않는다.

## 10.5 품질

- 전체 무역 루프를 3회 이상 반복한다.
- 성공·실패·이벤트·투자·대출·저장 실패 복구 시나리오를 수행한다.
- 동일 정산의 중복 보상과 중복 상환이 발생하지 않는다.
- 동일 투자 퀘스트의 중복 완료와 중복 해금이 발생하지 않는다.
- Console Error가 없다.
- Blocker와 Critical 결함이 없다.
- 독립 실행 빌드에서 핵심 기능이 동작한다.

# 15. 최종 구현 목표

최종 제출 시 플레이어는 다음 흐름을 안정적으로 경험할 수 있어야 한다.

```text
새 게임 또는 이어하기
→ 최대 4개의 Caravan 중 하나를 선택해 준비
→ 도시·경로·마차·동물·용병·식량·무역품 구성
→ 출발 검증과 저장 성공
→ Traveling 및 온라인·오프라인 진행
→ 계절·재난·이벤트·약탈 결과
→ 성공·실패·마차 파괴 정산
→ 구조 대출이 있다면 상환 여부 선택
→ Claim과 저장
→ 성장·수리·건축 재료 이전·건물 업그레이드
→ 거래 재화와 Caravan 무역품으로 투자 퀘스트 완료
→ 신규 콘텐츠 해금
→ 진행 불능 시 1회성 구조 대출로 최소 무역 재개
→ 구조 대출 사용 후 다시 회생 불가능하면 게임오버
→ 재실행 후 동일 상태 복구
```

핵심 품질 기준:

- 모든 중요 상태 변화는 저장 성공 후에만 완료로 확정한다.
- 저장 실패 시 성공 이벤트, 성공 UI, 다음 화면 전환을 발생시키지 않는다.
- 중요 Command는 실행 직전 snapshot으로 rollback할 수 있다.
- 정의 데이터는 Shared Data에서 조회하고 SaveData에는 플레이어 결과 상태만 저장한다.
- 무역과 정산은 `caravanId + full tradeId`로 식별한다.
- 건물은 표시 이름이 아닌 stable `buildingId`로 식별한다.
- 투자 퀘스트는 거점 도시 인벤토리를 제출 원천으로 사용하지 않는다.
- 구조 대출은 게임을 재개시키기 위한 1회성 구제 수단이며 일반 성장 자금이 아니다.

# 16. 비주얼 기획 및 화면 구성 개선 방향

## 16.1 문서 역할과 확정 범위

본 섹션은 프로젝트의 주요 기능 개발이 완료된 이후, 실제 구현 우선순위와 제작 여건에 맞춰 세부 정책을 구체화하기 위한 **비주얼 방향성 문서**다.

현재 단계에서 본 섹션은 다음 성격을 가진다.

- 제품의 장기적인 화면 구성 목표를 정의한다.
- UX·아트·화면 연출의 공통 방향을 제시한다.
- 실제 세부 정책, 구현 방식, 우선순위, 리소스 범위는 **핵심 기능 개발 완료 후** 별도 합의로 확정한다.
- 본 섹션의 내용은 즉시 구현 의무가 아니라, 이후 UI & Data / Content & Tools / Framework 통합 설계 시 참조할 상위 비전이다.

즉, 현재는 “어떤 화면 경험을 만들 것인가”를 정리하는 단계이며, 구체적인 씬 구조, 카메라 동작, 패널 시스템, 인터랙션 상세 규칙은 후속 비주얼 정책 문서에서 확정한다.

## 16.2 비주얼 목표 한 줄 정의

> 플레이어가 여러 개의 독립된 UI 창을 조작하는 게임이 아니라, 아이소메트릭 거점 도시에서 Caravan을 준비하고 월드맵의 무역로를 따라 교역을 운영하는 게임처럼 느껴지도록 구성한다.

## 16.3 무역 기능과 미니맵의 연계

무역 기능을 별도의 UI 패널 안에서만 처리하지 않고, 월드맵 또는 미니맵과 직접 연동하는 방향을 검토한다.

플레이어가 목적지 도시나 무역로를 선택하면 미니맵에서 해당 위치와 이동 경로가 강조되며, 무역 버튼은 현재 선택된 도시 및 경로 정보와 연결되어 동작하도록 구성한다.

무역 진행 중에는 미니맵을 통해 다음 정보를 직관적으로 확인할 수 있도록 한다.

- 현재 Caravan의 위치
- 출발 도시와 목적지 도시
- 이동 경로
- 무역 진행률
- 예상 도착 지점
- 경로상의 위험 요소 및 이벤트 발생 위치

이를 통해 무역 준비, 출발, 이동 진행이 서로 분리된 메뉴가 아니라 하나의 연속된 공간 경험으로 느껴지도록 한다.

## 16.4 세계관에 맞는 용어 재정의

현재 사용 중인 기능명과 재화 명칭을 게임의 중세 판타지 무역 콘셉트에 맞게 재검토한다.

예를 들어 일반적인 명칭인 `골드`, `돈`, `재화`를 그대로 사용하는 대신 세계관에 어울리는 명칭을 적용할 수 있다.

예시:

- 골드 → 금화
- 소액 화폐 → 은화
- 고급 또는 특수 화폐 → 두카트
- 발전용 재화 → 공헌도, 건설 증서, 도시 인장
- 투자 재화 → 개척 증서, 투자 증서
- 무역 시작 → Caravan 출정, 교역 출발
- 정산 → 교역 결산, 원정 결산

단순히 명칭만 변경하는 것이 아니라 UI 문구, 아이콘, 정산 화면, 상점 표시 방식까지 동일한 용어 체계를 유지한다.

용어의 일관성은 플레이어가 게임 세계를 하나의 실제 무역 사회로 인식하게 만들며, 전체적인 몰입감 향상에 기여한다.

## 16.5 아이소메트릭 거점 도시의 기능 확장

현재 아이소메트릭 뷰로 표현되는 거점 도시는 단순한 배경 화면이 아니라 게임의 주요 기능을 이용하는 중심 허브로 발전시킨다.

거점 도시 내의 건물과 시설을 직접 선택하여 각 기능에 접근할 수 있도록 구성한다.

예시:

- 시장 또는 상단 건물 → 상품 구매 및 Caravan 준비
- 마구간 → 견인 동물 구매·배치·관리
- 마차 제작소 → 마차 구매·수리·강화
- 용병 길드 → 용병 고용 및 전투력 확인
- 시청 또는 길드 회관 → 투자 퀘스트·구조 대출
- 지도 시설 또는 성문 → 무역로 및 목적지 선택
- 창고 → 보유 물품과 Caravan 적재 현황 확인

도시의 건물은 단순한 장식물이 아니라 각 시스템으로 진입하는 시각적 인터페이스 역할을 한다.

장기적으로는 건물의 해금, 성장, 투자 상태에 따라 도시 외형이 변화하도록 확장할 수 있다.

## 16.6 UI 패널 의존도 축소

현재 화면을 크게 차지하는 고정형 UI 패널의 비중을 줄이고, 거점 도시와 월드맵 자체를 주요 정보 전달 공간으로 활용한다.

핵심 정보만 화면 가장자리의 HUD 형태로 유지하고, 상세 정보는 플레이어가 건물이나 오브젝트를 선택했을 때 열리는 팝업 또는 슬라이드 패널로 제공한다.

권장 구성:

- 상단: 보유 재화, 현재 계절, 주요 알림
- 좌측 또는 우측: 선택된 건물·도시·Caravan의 상세 정보
- 하단: 주요 행동 버튼과 현재 진행 상태
- 중앙: 거점 도시 또는 월드맵을 최대한 넓게 표시

이를 통해 플레이어가 UI 창을 조작하는 느낌보다 실제 도시와 지도를 관리하는 느낌을 받을 수 있도록 한다.

단, 모든 기능을 도시 오브젝트 안에 숨기기보다는 자주 사용하는 기능에는 단축 버튼을 함께 제공하여 접근성을 유지한다.

## 16.7 거점 도시와 무역 준비 화면의 통합

무역 준비 과정을 하나의 대형 패널에서 처리하는 대신, 거점 도시의 여러 시설을 순차적으로 이용하여 Caravan을 구성하는 방식도 검토한다.

예시 흐름:

```text
시장 선택
→ 무역품과 식량 구매

마구간 선택
→ 견인 동물 배치

마차 제작소 선택
→ 마차 선택 및 수리

용병 길드 선택
→ 용병 고용

지도 또는 성문 선택
→ 목적지와 무역로 결정

출정 버튼 선택
→ 최종 구성 확인 후 무역 출발
```

각 시설에서 변경한 내용은 하나의 Caravan 준비 데이터에 누적되며, 화면 하단 또는 측면의 간단한 요약 UI를 통해 현재 구성을 계속 확인할 수 있도록 한다.

이를 통해 거점 도시가 단순한 메뉴 배경이 아니라 실제 플레이 루프의 중심 공간으로 기능하게 한다.

## 16.8 전체 화면 비율 및 해상도 대응

UI와 거점 도시 화면은 특정 해상도에서만 정상적으로 보이는 고정 배치 방식이 아니라, 다양한 화면 비율에 대응할 수 있도록 설계한다.

기본 기준 해상도는 `1920 × 1080`, 화면 비율은 `16:9`로 설정하되 다음 환경을 추가로 확인한다.

- 16:9
- 16:10
- 21:9
- 창 모드
- 낮은 해상도 환경

핵심 UI는 화면 가장자리의 Safe Area 안에 배치하고, 화면 비율이 달라지더라도 버튼, 텍스트, 자원 표시와 주요 건물이 잘리거나 겹치지 않도록 한다.

거점 도시와 월드맵은 화면 크기에 따라 단순 확대되는 방식보다 다음과 같은 대응을 우선한다.

- 카메라 표시 영역 확장
- UI 앵커와 레이아웃 그룹 적용
- 최소·최대 패널 크기 설정
- 긴 텍스트의 자동 줄바꿈
- 작은 해상도에서 일부 부가 정보 접기
- 울트라와이드 화면에서 빈 공간을 도시 배경과 지도 영역으로 활용

## 16.9 비주얼 레퍼런스 방향

비주얼과 정보 전달 방식의 참고 대상으로 다음 게임을 활용한다.

### 미니 메트로

참고 요소:

- 복잡한 정보를 간결한 선과 아이콘으로 표현하는 방식
- 노선과 이동 상태를 한눈에 파악할 수 있는 지도 구성
- 제한된 색상과 단순한 형태를 활용한 높은 정보 가독성
- 화면을 크게 가리지 않는 최소한의 UI

프로젝트에는 무역로, 도시 연결, Caravan 이동 경로, 위험 구간과 진행 상태를 간결하게 표현하는 방향으로 참고한다.

### 프로스트펑크

참고 요소:

- 도시 자체를 주요 인터페이스로 사용하는 구성
- 건물을 선택하여 기능에 접근하는 방식
- 도시 성장과 시스템 진행이 시각적으로 연결되는 구조
- 세계관에 맞는 용어, 아이콘, 색상과 UI 장식
- 화면 가장자리 UI와 중앙 도시 화면의 역할 분리

프로젝트에는 거점 도시의 시설 선택, 건물 성장 표현, 경제 시스템 진입 방식과 세계관 중심 UI 구성 측면에서 참고한다.

다만 대규모 도시 건설이나 복잡한 3D 연출을 그대로 구현하는 것이 아니라, 현재 프로젝트 규모에 맞게 단순화된 아이소메트릭 도시 허브와 건물 선택형 UI로 적용한다.

## 16.10 비주얼 설계 원칙

향후 세부 정책을 정할 때 다음 원칙을 우선 적용한다.

- 거점 도시와 월드맵을 화면의 중심으로 사용한다.
- UI는 핵심 정보와 선택 보조 역할에 집중한다.
- 건물, 도시, 경로와 주요 기능을 시각적으로 연결한다.
- 세계관에 맞는 재화와 기능 용어를 일관되게 사용한다.
- 무역 준비부터 출발, 이동, 정산까지 하나의 연속된 경험으로 표현한다.
- 화면 비율이 달라져도 핵심 플레이 정보와 조작 요소가 유지되도록 한다.

## 16.11 후속 정책 확정 시점

비주얼 관련 세부 정책은 다음 조건이 충족된 이후 확정한다.

- 플레이어용 핵심 무역 루프가 기능적으로 안정화되었을 것
- 주요 저장·복구·정산·성장 시스템의 데이터 흐름이 고정되었을 것
- 화면 구조 변경이 공용 API나 SaveData 구조를 다시 흔들지 않는 시점일 것

후속 확정 대상 예시:

- 미니맵과 월드맵의 실제 인터랙션 규칙
- 거점 도시 카메라 동작과 선택 방식
- 건물 선택 UI와 상세 패널 전개 방식
- 용어 최종안과 아이콘 체계
- HUD / 팝업 / 슬라이드 패널의 우선순위와 정보 밀도
- 해상도 대응용 레이아웃 규칙과 최소 지원 사양
- 실제 제작 가능한 비주얼 리소스 범위와 단계별 적용 우선순위

본 섹션은 이후 비주얼 정책 문서, UI 플로우 문서, 와이어프레임, 컨셉 시트의 상위 기준으로 사용한다.
