# 누워서 돈벌기 — 2차 빌드 팀 공통 마일스톤 및 운영 규칙

## 1. 문서 목적

본 문서는 2026-07-20 업데이트 기획서를 기준으로 5인 팀의 2차 빌드와 최종 빌드 작업을 재배치한다.

기존 독립 기부 시스템은 공식 범위에서 제거하며, 투자 퀘스트와 1회성 구조 대출을 중심으로 경제·성장·복구 시스템을 완성한다. 비주얼 및 화면 구성 개선안은 핵심 기능 개발 이후 세부 정책을 정하는 후속 단계로 분리한다.

## 2. 담당자

| 담당자 | 역할 |
|---|---|
| 천성욱 | Framework & Integration, 팀 통합, SaveData, 저장 복구 |
| 윤호영 | Core Gameplay, Caravan·Journey·손실·정산 런타임 |
| 박준서 | UI & Data, Scene Owner, 제품 UI·화면 연결 |
| 정정헌 | Progression & System, 경제·성장·수리·대출 계산 |
| 이종현 | Content & Tools, Shared Data·콘텐츠·검증 도구 |

## 3. 공통 일정 원칙

- **2차 빌드 개발 기간:** 2026-07-20 ~ 2026-08-06
- **최종 빌드 개발 기간:** 2026-08-07 ~ 2026-08-11
- **기준 브랜치:** `dev2`
- **통합 방식:** 기능 브랜치 → PR → `dev2`
- **구조 동결:** 2026-08-04 업무 종료
- **기능 동결:** 2026-08-05 업무 종료
- **2차 빌드 RC:** 2026-08-06
- **1차 빌드 고정:** 1차 빌드 범위와 통과 조건은 소급 수정하지 않는다.
- **비주얼 세부 정책:** 핵심 기능과 데이터 흐름 안정화 후 별도 합의로 확정한다.


## 4. 2차 빌드 최종 목표

```text
Caravan 선택·준비
→ 목적지·경로·상품·식량·마차·동물·용병 구성
→ 출발 검증과 즉시 저장
→ Traveling·오프라인 복구
→ 이벤트·손실·마차 파괴
→ SettlementPending·선택 상환·Claim
→ 성장·수리·거점 인벤토리·건물 업그레이드
→ 투자 퀘스트 완료·콘텐츠 해금
→ 진행 불능 시 1회성 구조 대출
→ 재실행 후 동일 상태 복구
```

## 5. 추가 구현 큰 기능 단위와 담당 일감

### 5.1 멀티 Caravan 준비·운영

- **윤호영 — Core Gameplay**
  - Caravan별 준비·Traveling·SettlementPending 런타임 상태 분리
  - 한 Caravan당 활성 무역 1개 검증
  - Traveling·SettlementPending Caravan 자산 잠금 검증
- **천성욱 — Framework & Integration**
  - Caravan 컬렉션 SaveData와 `caravanId + full tradeId` 식별 계약
  - 마지막 선택 Caravan과 준비 데이터 저장·복구
  - 최대 수를 SaveData 고정값이 아닌 Shared Data/Progression 설정으로 조회
- **박준서 — UI & Data**
  - Caravan 선택·전환 UI
  - 선택 Caravan별 준비 정보·진행 상태·정산 대기 표시
- **정정헌 — Progression & System**
  - Caravan 해금 수와 최대 보유 수 정책
  - 자산 사용 가능성·경제 상태 판정 규칙
- **이종현 — Content & Tools**
  - Caravan 초기 프리셋과 ID 검증 데이터
  - 복수 Caravan 테스트 데이터 제작

### 5.2 무역 준비·출발·Traveling·정산 복구

- **윤호영**
  - 상품·식량·마차·견인 동물·용병 구성 검증
  - 슬롯·적재·과적·최소 견인 수 검증
  - 성공·실패·정산 snapshot 생성과 중복 Claim 방지
- **천성욱**
  - 출발·정산 Claim 중요 Command 저장 transaction
  - Traveling·SettlementPending 재실행 복구
  - 저장 성공 이후 이벤트·UI 갱신 순서 보장
- **박준서**
  - 제품용 Preparation·Traveling·Settlement UI
  - 출발 차단 사유, 진행률, 손실, 순이익, Claim 표시
- **정정헌**
  - 구매·판매·식량·용병 비용과 정산 순이익 계산
  - 거리·과적·식량·위험도 계산 ViewData 제공 규칙
- **이종현**
  - 도시·경로·상품·마차·동물·용병 데이터 입력
  - 경계값과 반복 무역 테스트 프리셋

### 5.3 저장 안정성·SaveResult·Snapshot·Rollback

- **천성욱**
  - `SaveResult Save(SaveData data)` 목표 API
  - PreCommandSnapshot·LastDurableSnapshot·Disk Backup 구분
  - 임시 파일→검증→메인 교체, 최대 3회 재시도
  - rollback 성공·실패와 저장 실패 이벤트 계약
- **윤호영**
  - Core aggregate를 snapshot 복원 가능한 구조로 제공
  - rollback 후 파생값 재계산과 런타임 상태 재구성
- **박준서**
  - 저장 처리 중 기능 패널 입력 차단
  - 실패 이유 메시지 키 매핑, rollback 후 ViewData 재조회
- **정정헌**
  - 중요 Progression Command 결과와 저장 실패 구분
  - 향후 `CommandResult` 확장에 필요한 도메인 실패 이유 정리
- **이종현**
  - 손상·누락·구버전·재시도 테스트 데이터와 검증 도구

### 5.4 성장·마차 내구도·파괴·수리

- **윤호영**
  - 내구도 감소, 0 clamp, 마차 파괴와 무역 즉시 실패
  - 파괴 시 wagon 참조·적재 무역품·식량 제거
  - 정산 snapshot에 파괴·손실·실패 사유 기록
- **천성욱**
  - 파괴·손실·실패·snapshot 저장을 단일 staged operation으로 연결
  - 저장 실패 시 파괴 이전 상태 rollback
- **박준서**
  - 내구도·수리량·수리 비용·파괴 결과 UI
  - 파괴된 마차 선택 차단과 구매 유도
- **정정헌**
  - `floor(수리량 × 단가 × 레어리티 배율)` 계산
  - 최소 비용 1, 최대 내구도 초과·파괴 마차 수리 차단
- **이종현**
  - 마차별 수리 단가·레어리티 배율·내구도 데이터
  - 파괴·부분 수리·최대 수리 테스트 프리셋

### 5.5 거점 도시 인벤토리·건축 재료·건물 업그레이드

- **정정헌**

  - 거점 인벤토리만 업그레이드 재료로 사용하는 검증
- **이종현**

## 8. 완료 기준

- 제품 UI만으로 전체 루프를 3회 이상 반복한다.
- Traveling·SettlementPending·투자·대출·건물·수리 상태가 재실행 후 유지된다.
- 모든 중요 Command는 저장 성공 후에만 완료된다.
- 저장 실패 시 성공 이벤트·화면 전환·보상이 발생하지 않는다.
- 독립 기부·누적 투자 진행도·이자 대출 정책이 잔존하지 않는다.
- 비주얼 세부 정책은 핵심 기능 완료 뒤 별도 합의 대상으로 유지된다.
