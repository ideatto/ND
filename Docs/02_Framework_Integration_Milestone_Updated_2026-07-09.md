**최종 마감:** 2026년 7월 16일 오전 10시  
**기능 동결:** 2026년 7월 14일 오후 6시  
**문서 업데이트:** 2026년 7월 9일 — 저장 안정성·자동저장·Caravan 시간 기반 식량·오프라인 시간 배율 범위 반영

### 해야 할 일

- Boot, Title, Loading, InGame 씬 연결
- 공용 매니저 중복 생성 방지
- 새 게임과 이어하기 분기 인터페이스
- 게임 시간 서비스
- 현실 경과 시간을 인게임 경과 시간으로 변환하는 배율 정책
- 온라인·오프라인 공통 경과 인게임 시간 제공 초안
- 저장 서비스 인터페이스
- 저장 데이터 버전 필드
- 공용 로그
- 시간 배속
- 무역 즉시 완료
- 통합용 샘플 씬

### 완료 기준

- 씬 흐름이 처음부터 끝까지 동작한다.
- 공용 매니저가 중복 생성되지 않는다.

- 각 담당자가 시간과 저장 API를 호출할 수 있다.

### 필요한 협업

- UI & Data: `SaveData` 정의
- Core Gameplay: 저장해야 할 Caravan 원본 상태와 계산값 제외 기준
- Core Gameplay: 식량 계산에 전달할 경과 인게임 시간 형식
- Progression: 계절·재난 시간 요구사항
- Content: 테스트 데이터 초기화

## M3 — 7월 13일 오후 10시

### 해야 할 일

#### 기본 저장과 오프라인 진행

- JSON 저장과 불러오기
- 저장 데이터 버전 기록
- 저장 순번(`saveSequence`) 증가와 저장 시각 기록
- 종료 시각 저장
- 오프라인 경과 시간 계산
- 진행 중 무역 남은 시간 복구
- 오프라인 현실 경과 시간을 동일한 인게임 시간 배율 정책으로 변환
- 변환된 경과 인게임 시간을 Core 진행도·식량 소모 복구에 전달
- 완료 무역 정산 대기 처리
- 정산 수령 플래그
- 중복 보상 방지
- 시간 역행 감지
- 최대 오프라인 인정 시간
- 기부·투자·대출·튜토리얼 상태 저장
- 마차 ID·현재 내구도·견인 동물 ID와 수·화물·식량 등 Caravan 원본 상태 저장
- 최종 적정 적재량·과적 비율·최종 이동속도는 저장하지 않고 로드 후 재계산

#### 저장 안정성 및 복구

- 현재 메인 저장 파일에 직접 덮어쓰지 않는 저장 흐름 구현
- 새 데이터를 임시 파일에 먼저 기록
- 임시 파일을 다시 읽어 역직렬화와 유효성 검사 수행
- 검증된 임시 파일만 메인 파일로 교체
- 교체 전 기존 정상 메인 파일을 백업으로 보존
- 실행 시 메인·임시·백업 파일을 모두 검사
- 유효한 후보 중 저장 순번이 가장 높은 파일을 로드
- 손상된 파일을 즉시 삭제하거나 기본 데이터로 덮어쓰지 않음
- 모든 후보가 유효하지 않을 때만 새 게임 전환 결과를 UI에 전달
- 복구에 사용된 파일 종류와 실패 원인을 로그에 기록

#### 자동저장

- 영속 데이터 변경 시 Dirty Flag 설정
- 10초마다 Dirty 상태를 확인하고 변경이 있을 때만 저장
- 무역 출발, 정산 완료, 성장 구매, 기부, 투자, 대출 확정 시 즉시 저장 요청
- 임시 UI 선택, 미리보기, 취소 가능한 입력은 즉시 저장 대상에서 제외
- 짧은 시간에 반복된 저장 요청을 하나로 병합
- 동시에 두 저장 작업이 실행되지 않도록 직렬화
- 저장 중 추가 요청이 발생하면 완료 후 최신 상태를 한 번 더 저장
- 씬 전환 전 저장
- 애플리케이션 일시정지와 정상 종료 시 저장 시도
- 종료 콜백이 호출되지 않는 상황을 전제로 중요 상태 전환 저장을 우선 적용

### 완료 기준

- 새 게임과 이어하기가 모두 동작한다.
- 게임 종료 후 상태를 복구한다.
- 메인 저장이 손상돼도 정상 백업 또는 임시 파일로 복구한다.
- 유효한 후보가 여러 개면 저장 순번이 가장 높은 데이터를 선택한다.
- 모든 저장 후보가 손상된 경우에만 새 게임 전환을 안내한다.
- 손상된 저장 파일을 자동으로 기본 데이터로 덮어쓰지 않는다.
- 변경 사항이 없으면 10초 주기에서 파일을 다시 쓰지 않는다.
- 중요 상태 전환은 10초 주기를 기다리지 않고 저장된다.
- 저장 요청이 연속 발생해도 파일 쓰기가 동시에 실행되지 않는다.
- 오프라인 완료 보상이 중복되지 않는다.
- 오프라인 경과 시간 배율·식량 소모 복구 결과가 중복 적용되지 않는다.
- 오프라인 식량 소모가 온라인과 동일한 인게임 시간 변환 규칙을 사용한다.
- 저장·로드 전후 최종 적정 적재량과 과적 비율이 Core 재계산 결과와 일치한다.
- 비정상 시간 역행을 감지한다.
- 저장 데이터 버전·저장 순번·저장 시각이 기록된다.

### 일정 지연 시 축소

- 서버 시간 검증 제외
- HMAC과 암호화는 2차 빌드로 이관
- 자동저장은 10초 Dirty 저장과 중요 상태 전환 저장만 유지
- 복구 후보는 메인·백업을 필수로 하고, 임시 파일 복구는 가능하면 유지
- 마이그레이션은 현재 버전 하나만 지원
- 시간 조작 방어는 역행 감지와 상한만 유지

- [ ] 변경이 없는 상태에서는 자동저장이 불필요한 파일 쓰기를 하지 않는다.
- [ ] 중요 상태 전환은 즉시 저장된다.
- [ ] 여러 저장 요청이 발생해도 저장 작업이 동시에 실행되지 않는다.
- [ ] 저장 실패가 로그와 UI에 전달된다.
- [ ] 메인·임시·백업 중 최신 유효 데이터를 복구한다.
- [ ] 손상된 저장 파일을 자동 삭제하거나 기본 데이터로 덮어쓰지 않는다.
- [ ] 모든 저장 후보가 유효하지 않을 때만 새 게임 전환을 안내한다.
- [ ] 제출용 빌드와 소스 커밋을 연결할 수 있다.

# 6. 2차 빌드 후속 마일스톤 — 저장 데이터 보호

> 이 범위는 1차 빌드 저장 안정성·복구가 검증된 이후 진행한다. 1차 빌드 M3에는 HMAC 또는 암호화를 포함하지 않는다.

## 목표

- 저장 데이터 변조 감지를 위한 HMAC 적용
- HMAC 검증 실패 데이터를 유효하지 않은 저장 후보로 처리
- 기존 메인·임시·백업 복구 흐름과 결합
- 보호 계층을 직렬화와 파일 저장 계층에서 분리

## 해야 할 일

- JSON 직렬화 결과를 보호할 `SaveDataProtector` 계약 정의
- HMAC 생성과 검증 구현
- 저장 시 payload와 인증값을 함께 기록
- 로드 시 역직렬화 전에 HMAC 검증
- HMAC 불일치 시 다른 메인·임시·백업 후보 검사
- HMAC 키를 코드의 평문 문자열 리터럴로 직접 저장하지 않음
- 대상 플랫폼에 맞는 보안 저장소, 빌드 시 주입 또는 분리된 키 관리 계층 검토
- 키 누락·변경·플랫폼 이전 시의 실패 처리 정의
- 기존 1차 빌드 저장 데이터의 전환 또는 초기화 정책 문서화
- 클라이언트 HMAC은 일반적인 파일 편집을 어렵게 하지만 완전한 안티치트가 아님을 문서화

## 완료 기준

- 저장 payload를 수정하면 HMAC 검증에 실패한다.
- 정상 저장은 기존 복구 흐름을 통해 문제없이 로드된다.
- 한 후보의 HMAC이 실패해도 다른 정상 백업 후보를 검사한다.
- 직렬화, 데이터 검증, HMAC 보호, 파일 저장을 독립적으로 테스트할 수 있다.
- 저장소 소스 코드에 읽을 수 있는 평문 HMAC 키가 커밋되지 않는다.
